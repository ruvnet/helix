# ADR-017: Mint Helix as a Branded Harness via MetaHarness

**Status**: Proposed
**Date**: 2026-06-25
**Project**: Helix — Personal Health Intelligence (PHI)
**Prepared by**: ISO Vision LLC
**Substrate**: Ruflo + RuVector + Cognitum Seed + MetaHarness/Darwin
**Related**: ADR-002, ADR-001, ADR-013, ADR-018, ADR-019, ADR-103

---

## Context

### The fork problem and why Ruflo alone is not enough

Ruflo is the orchestration meta-harness that powers Helix (ADR-002). It provides multi-agent
coordination, swarm consensus, SONA learning loops, AIDefence PII-gating, and HIPAA-mode audit
trails. But Ruflo is a general-purpose substrate — it does not carry Helix's identity, default-deny
security posture, domain-scoped memory namespace, or the governance wrapper that a health product
demands. The naive path is to fork Ruflo, add Helix-specific content, and ship the fork as the
product. That path breaks every time the Ruflo or RuVector kernel ships a meaningful update: the fork
diverges, and keeping it current requires manual rebasing work that compounds over time.

Evidence grade: **[A]** — observed directly in every health-adjacent software product that forked a
general agent framework (Langchain, AutoGPT-adjacent variants, OpenAgents). The pattern is
consistent: the fork stagnates relative to the upstream kernel within 3–6 months of divergence.

### What MetaHarness is — and what it is not

`metaharness` (npm package v0.1.x, repo `ruvnet/agent-harness-generator`, author ruv@ruv.net) is a
**harness generator** — a CLI/factory that points at a repo and emits a branded, npm-publishable
agent CLI. The generator's tagline captures the core architecture principle: *"The model is
replaceable. The harness is the product."* **[A]** — directly observed from the package on npm and
its GitHub repository at `ruvnet/agent-harness-generator`, confirmed by local repo investigation
(ADR-256 research artifact at `docs/adr/ADR-256-metaharness-sdk-evaluation.md`).

MetaHarness is **not** a runtime SDK that Helix embeds as a production dependency. This distinction
is load-bearing. ADR-150 in the ruvector repository establishes MetaHarness as a *removable
augmentation*: every MetaHarness feature must degrade gracefully when the `@metaharness/*` packages
are absent, every Darwin call must be wrapped in try-catch with a grid-search fallback, and a CI
gate must pass without MetaHarness installed. ADR-256 further establishes that the dependency
direction runs *toward* ruvector (`metaharness` optionally depends on `@ruvector/ruvllm` and links
`@ruvector/emergent-time`), making a runtime dependency circular. **[A]** — grounded in local ADR
files and package dependency graph analysis.

The generator supports named vertical templates. The `vertical:coding` template is demonstrated in
the live timesfm-harness at `harnesses/timesfm-harness/` in this repo — generated by
`create-agent-harness` v0.2.7, kernel 0.1.2, yielding a four-agent engineering pod (architect,
implementer, reviewer, test-writer) with default-deny MCP, a `.harness/manifest.json` provenance
record, and an `evolve` Darwin Mode skill. **[A]** — directly read from
`harnesses/timesfm-harness/PROVENANCE.md` and `harnesses/timesfm-harness/.harness/manifest.json`.

A `vertical:health` template — the appropriate archetype for Helix — would apply the same factoring
to a health-domain harness, scoped to the agent roster described in the Helix product spec (§3).
**[B]** — the `vertical:health` template is referenced in the product spec and in §13 of the
Helix PHI ADR spec; existence of the template in the generator is not independently confirmed at
document date; treat as design intent pending generator version confirmation.

### The "minted, not forked" pattern — how it solves the divergence problem

MetaHarness separates:

1. **The kernel** — the reusable, domain-agnostic Ruflo/RuVector runtime (versioned in
   `package.json`, pinned but upgradeable via semver range).
2. **The harness** — the opinionated, domain-specific layer: the agent roster, the CLI identity,
   the MCP tool allowlist, the memory namespace, the governance policy, the skill surface, and the
   evolve/Darwin configuration.

The kernel is pulled as a dependency. The harness is what the generator mints. When the kernel
ships a security patch or a performance improvement, `npm update` captures it without touching
harness-specific content. The harness owner decides what kernel version to accept — "pinned but
upgradeable" — and can review kernel changelogs before bumping.

Evidence grade for this architectural pattern: **[A]** — grounded in the timesfm-harness example
(live in repo), the ADR-256 analysis, and the generator's own documentation. The "vertical:health
template" specifics are **[B]** pending confirmed generator support.

### What Helix needs that a bare Ruflo configuration does not provide

1. **A distinct CLI identity.** `npx helix` or `npx @helix-phi/cli` must be a real, publishable
   npm artifact with its own version history, changelog, and release integrity guarantees —
   not an alias for `npx ruvector` with a flag.

2. **Default-deny MCP.** The MCP tool surface for a health product must be closed by default: only
   explicitly allowed tools are available to agents. This prevents prompt injection from expanding
   the accessible tool surface. The timesfm-harness example demonstrates the pattern: the generated
   `.claude/settings.json` carries a default-deny allowlist. **[A]** — read from
   `harnesses/timesfm-harness/.claude/settings.json` hash in the manifest.

3. **Scoped memory namespace.** Helix's agents write to and read from a `helix` memory namespace
   in RuVector's AgentDB. Cross-contamination with unrelated projects' SONA patterns is a data
   hygiene and privacy risk. The minted harness encodes the namespace in its configuration so every
   agent spawned by the harness uses it by default without ad-hoc convention.

4. **Ed25519 witness-signed releases.** Health-context software must be supply-chain-clean. The
   generator emits a `.harness/manifest.json` that records a SHA-256 of every emitted file and a
   witness hash over the manifest. For Helix releases, this witness must be extended to a full
   Ed25519 signature, giving third parties a verifiable audit trail from source commit to shipped
   harness artifact. This builds on the ruvector witness chain established in ADR-103. **[A]** —
   grounded in timesfm-harness PROVENANCE.md and ADR-267 (SOTA validation protocol).

5. **Health-domain governance policy.** The harness-level CLAUDE.md encodes rules that apply to
   every agent in every session: no diagnostic assertions (ADR-010), evidence-tiering on every
   recommendation (ADR-006), abstention when data is absent (ADR-005), red-flag escalation routing
   to the Guardian (ADR-009), and explicit "augments, does not replace, a clinician" framing.
   These are not best-effort prompt conventions; they are harness-level constraints enforced before
   any agent prompt executes.

6. **Kernel pinned but upgradeable.** The harness's `package.json` pins `@metaharness/kernel` to
   a semver range (e.g., `^0.1.2`) and the Ruflo/RuVector substrate packages to minor-version
   ranges. Helix can review upstream releases before bumping — preventing silent breaking changes
   in medical-context software — while still pulling security patches without manual rebasing.

## Decision

Helix is generated from the MetaHarness `vertical:health` template as a branded harness. The
generator is run once to produce the harness scaffold; the output is version-controlled in the
Helix repository. The kernel is a versioned dependency, not vendored source.

### The `vertical:health` mint

```bash
# Feasibility scorecard (read-only, no changes)
node create-agent-harness/dist/bin.js score /path/to/helix-repo --json

# Genome analysis (repo synthesis verdict)
node create-agent-harness/dist/bin.js genome /path/to/helix-repo --json

# Mint the harness
node create-agent-harness/dist/bin.js helix-phi \
  --template vertical:health \
  --host claude-code \
  --description "Helix PHI — Personal Health Intelligence: private, local-first, anti-hallucination health dossier" \
  --target ./
```

The generator emits:

| Artifact | Purpose |
|---|---|
| `.harness/manifest.json` | Schema 1, template/vars/hosts, SHA-256 per emitted file, generated_at |
| `.harness/manifest.sha256` | Witness hash over the manifest (extend to Ed25519 for releases) |
| `.claude/settings.json` | Default-deny MCP allowlist — only explicitly listed tools accessible |
| `.claude-plugin/plugin.json` | Plugin registration for the harness's MCP surface |
| `CLAUDE.md` | Harness-level behavioral rules (health governance, evidence tiering, abstention policy) |
| `bin/cli.js` | The branded `helix` CLI entry point |
| `package.json` | `@metaharness/kernel` in optionalDependencies, Ruflo substrate pinned |
| `src/agents/*.ts` | Domain specialist agents (see agent roster below) |
| `.claude/skills/evolve/SKILL.md` | Darwin Mode skill (ADR-018) |

### Agent roster

The `vertical:health` harness emits the agent roster described in the product spec §3, with
explicit model-tier assignments:

| Agent | Tier | Role |
|---|---|---|
| `ingestion` | sonnet | Per-connector, fault-tolerant, idempotent import |
| `normalization` | sonnet | LOINC/RxNorm/SNOMED/UCUM/FHIR mapping; provenance attachment |
| `fm-analyst` | sonnet | Functional-medicine reasoning from grounded facts |
| `verifier` | opus | Re-derives every analyst claim from source; gates output |
| `escalation-guardian` | opus | Red-flag threshold monitoring and urgent-care routing |
| `trend-numeric` | sonnet | Deterministic computation of slopes, deltas, correlations |
| `ambient-sensing` | sonnet | On-device mmWave signal extraction and anomaly detection |
| `experiment-planner` | sonnet | GOAP n-of-1 experiment design and tracking |
| `visualization` | sonnet | 3D twin, 0–100 score, trend rendering |
| `pii-gate` | haiku | AIDefence PII scan on any egress path |

The verifier and any DRACO-fitness judge agents are assigned to a *different model family* than the
fm-analyst synthesizer. This fusion principle (grounding verifier in a distinct family from the
synthesizer) is the architectural enforcement of the DRACO faithfulness requirement from ADR-018.

### Default-deny MCP

The generated `.claude/settings.json` begins with an explicit allowlist. No MCP tool is accessible
unless it appears in the list. Health-relevant egress tools (external search, network fetch,
cross-user memory queries) require explicit inclusion with documented justification. The allowlist is
reviewed and extended only through the Helix governance process (PR + clinical governance sign-off
for tools that expose health data).

### Scoped memory namespace

All harness-managed memory operations use the `helix` namespace in RuVector's AgentDB. SONA
trajectory recordings, ReasoningBank patterns, and the Darwin evolution archive all target this
namespace. The kernel configuration file encodes the namespace so it cannot be accidentally omitted
by individual agents.

### Ed25519 witness-signed releases

Release workflow:

1. `npm run build` produces the harness artifact.
2. The generator's witness mechanism creates `.harness/manifest.json` with SHA-256 per file.
3. A release signing step signs the manifest with the Helix Ed25519 private key (held in GCP Secret
   Manager, same pattern as ruvector's COGNITUM_AUTH_TOKEN / TAILSCALE_AUTH_KEY).
4. The public key is published in the repository at `docs/validation/helix-witness-public-key.pem`.
5. Any consumer can verify: `openssl pkeyutl -verify -pubin -inkey helix-witness-public-key.pem
   -sigfile manifest.sig -in manifest.json` before running a Helix harness release.

The signed manifest hash is committed to the release tag. This gives users a tamper-evident proof
that the harness they downloaded matches what was built and signed by ISO Vision LLC. This is the
same pattern used for ruvector SOTA validation manifests (ADR-267). **[A]**

### Kernel versioning policy

```json
{
  "optionalDependencies": {
    "@metaharness/kernel": "^0.1.2"
  },
  "dependencies": {
    "@ruvector/core": "^0.2.x",
    "@claude-flow/cli": "^latest-pinned"
  }
}
```

Kernel bumps require a changelog review. Major-version bumps require a migration review and
re-signing. The harness CI runs a `npm ls --without-deps @metaharness/*` gate to confirm the
harness functions without the optional kernel installed (ADR-150 invariant 1).

## Alternatives Considered

### Alternative 1: Fork Ruflo and ship a custom build

Fork the Ruflo open-source repository, add Helix-specific content, and publish a private fork.
The fork captures upstream periodically via manual rebase.

**Why rejected.** The fork divergence problem is well-documented: within 3–6 months of the fork,
upstream security patches and protocol changes require progressively expensive rebasing. For a
health product in a fast-moving substrate, this is a safety-critical maintenance debt. The fork
approach also loses the upstream kernel's semantic versioning guarantees — breaking changes arrive
without notice if the rebase is not vigilant. The "minted, not forked" pattern is specifically
designed to solve this problem without incurring the cost.

### Alternative 2: Operate Helix as a configuration layer on top of bare Ruflo

Configure Ruflo with a CLAUDE.md, a `.claude/settings.json`, and a custom agent roster, without
using the MetaHarness generator. Publish the configuration as a thin wrapper.

**Why rejected.** This approach lacks the provenance guarantees (witness-signed manifest), the
structured kernel-version pinning, and the generator-enforced invariants (default-deny MCP,
scoped namespace, evolve skill). It also lacks a publishable CLI identity. The result is
configuration drift: convention-based rules in CLAUDE.md are advisory, not enforced. For a health
product, advisory governance is insufficient.

### Alternative 3: Build a fully custom harness from scratch

Write all of the above — CLI, agents, MCP policy, signing, memory scoping, Darwin skill — without
the MetaHarness generator.

**Why rejected.** Engineering cost is non-trivial and the domain-specific value of Helix is in the
health-intelligence layer, not in harness infrastructure. The generator reduces the time-to-first
minted harness from weeks to hours. For a v0.1.x tooling dependency, the risk is tool maturity
(pre-1.0 beta); this is mitigated by the ADR-150 removability invariant — if MetaHarness stalls,
the generated output is fully self-contained and does not require the generator at runtime.

## Consequences

### Positive

- Helix has its own identity, CLI, and release cadence without forking the upstream kernel.
- Default-deny MCP is enforced at the harness level — every agent session begins with a closed
  tool surface.
- Health-domain governance rules are harness-level constraints, not prompt-engineering hopes.
- Ed25519 witness-signed releases give users and auditors a tamper-evident supply-chain record.
- Kernel upgrades are incremental: security patches arrive via `npm update` without touching
  Helix-specific harness content.
- The scoped memory namespace prevents cross-contamination between Helix health data and
  unrelated projects sharing the same AgentDB instance.

### Negative

- `metaharness` is pre-1.0 beta (v0.1.x). The generator API and template surface may change in
  breaking ways. Mitigation: the generated output is version-controlled; the generator is a
  build-time tool, not a runtime dependency. Breaking changes in the generator affect the
  next mint, not the current deployment.
- `vertical:health` template existence requires confirmation against the generator's current
  release. If the template is not yet available, the first step is contributing or scaffolding it
  before minting. Mitigation: use `vertical:coding` (confirmed available) as the starting point
  and overlay health-specific agents and governance rules manually; re-mint with `vertical:health`
  when available.
- Ed25519 signing adds a release-process step. Mitigation: automate in CI/CD alongside the
  existing NAPI-RS binary build workflow.
- The `@metaharness/kernel` optional dependency carries `@ruvector/emergent-time` as a transitive
  link. If `emergent-time` undergoes a breaking change, the kernel update must be reviewed before
  bumping. Mitigation: pin kernel to `^0.1.2` and review emergent-time changelogs on every kernel
  minor bump.

### Mitigations summary

| Risk | Mitigation |
|---|---|
| Generator pre-1.0 API instability | Generated output is committed; generator is build-time only |
| `vertical:health` template missing | Fall back to `vertical:coding` + manual overlay |
| Kernel breaking changes | Semver pin `^0.1.x`; CI gate without MetaHarness |
| Signing adds release overhead | Automate in CI alongside binary builds |

## Open Questions

1. **Template availability.** Does the `create-agent-harness` CLI (v0.2.7+) include
   `vertical:health`? If not, what is the contribution/scaffolding path?

2. **Kernel maturity gate.** At what kernel version (presumably `>=1.0.0`) should the
   `optionalDependencies` pin be promoted from `^0.1.x` to a stable major range? What is the
   upstream roadmap for `@metaharness/kernel` 1.0?

3. **Default-deny baseline.** Which MCP tools belong on the initial default-deny allowlist for
   a health harness? The list must be reviewed with clinical governance before the Phase 0
   harness ships.

4. **Key management.** Ed25519 signing requires a key ceremony: where are the private keys held
   (GCP Secret Manager, hardware token), who has signing authority, and what is the revocation
   path if the key is compromised?

5. **Multi-device vault and harness portability.** A user switching from phone to desktop must
   be able to run the same harness against the same encrypted vault. Does the harness's memory
   namespace scoping survive a vault migration or cross-device sync without data loss?
